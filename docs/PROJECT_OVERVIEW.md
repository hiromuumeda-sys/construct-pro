# construct-pro（WIN WIN様デモ）システム概要

提案書作成用の全体資料。2026年時点のコードベース実態に基づく。

---

## 1. プロジェクト概要

- **名称**: construct-pro（社内呼称：WIN WIN様デモ）
- **対象業種**: 建設・工事会社向け業務管理システム
- **性質**: デモ／PoC環境（本番運用前提の商用システムではなく、要件検証・提案用の動くデモ）
- **リポジトリ**: `~/construct-pro`（GitHub連携、Vercelに自動デプロイ）
- **稼働環境**: Vercel（本番）／ローカルNode.js（開発）

### カバーする業務領域
案件（受注）管理 → 工事計画（発注）管理 → 支払管理 → 売上・入金管理 → 帳票発行（見積書・発注書・請求書・注文請書）→ 履歴監査 → アカウント発行、までの一連の建設業務フローをワンストップで扱う。

|---|---|

## 4. データモデル（主要テーブル）

```
users             ── アカウント（email, password_hash, role）
audit_logs        ── 全操作の監査ログ（誰が・何を・いつ・差分）
invitations       ── アカウント招待（24時間有効トークン、role指定）

projects          ── 案件（顧客情報、契約金額、工期、ステータス）
orders            ── 工事計画＝発注明細（project紐付、金額3種、支払状況、残金、請書/請求書済フラグ）
order_files       ── 請書/請求書PDF本体（order_id×kind=ack|invoice、base64保存）
payment_records   ── 支払登録明細（消込履歴、残金への反映元）

vendors           ── 発注先マスタ（銀行口座情報含む）
customers         ── 顧客マスタ
categories        ── 工事区分マスタ

receipts          ── 入金記録
invoices          ── 発行済み請求書の永続化記録
```

**特記事項**
- `orders.remaining`：支払の残金。新規登録時は決定金額と同額を初期値とし、支払登録のたびに減算。0になると自動的に「支払済み」に遷移。
- `order_files`：請書・請求書PDFを**base64文字列としてDBに直接保存**する簡易方式（後述の課題を参照）。
- テーブルは起動時に `CREATE TABLE IF NOT EXISTS` で冪等に作成される簡易マイグレーション方式（`ensureAux()`）。正式なマイグレーション管理ツールは未導入。

## 5. 認証・権限設計

### 現状
- **認証**：自前実装。`bcrypt`でハッシュ化したパスワードを`users`テーブルに保存し、ログイン時に`jsonwebtoken`でJWT発行。`auth.js`がフロントの全fetchにトークンを自動付与。
- **権限（role）**：`admin`（管理者）／`accounting`（経理部）／`staff`（一般社員）の3種類。**現状は表示・招待発行にのみ使用しており、画面アクセスや機能の制御（RBAC強制）は未実装。**
- **招待フロー**：管理者が権限を指定してメール招待 → 24時間有効なトークン付きURL → 受諾者がパスワード設定してアカウント作成。

### Supabase Authとの関係（検討経緯）
- 現行DBはSupabase PostgreSQLを**ホスティングとしてのみ**利用。Supabase Auth・RLS（Row Level Security）は未使用。
- **RLSを使わない理由**：本アプリはフロントエンドが直接DBを叩く構成ではなく、必ずExpress API（サーバー）を経由するため、RLSが本来守る「クライアント→DB直結の穴」が存在せず、導入しても機能を持て余す。
- **今後の検討方針**：認証（サインアップ・ログイン・パスワードリセット・招待メール基盤）はSupabase Authに任せ、権限（画面/機能単位のRBAC）は引き続き自前実装（Expressミドルウェア）で持つ、というハイブリッド構成が有力候補。JWTにroleクレームを埋め込む「Custom Access Token Hook」で連携可能。

## 6. 帳票（PDF）機能

| 帳票 | 生成方式 | デザイン |
|---|---|---|
| 見積書・請求書 | `pdfkit`で明細表形式生成 | 顧客提供のExcelテンプレートに準拠（配色：紫 `#7030A0`、請求№・適格請求書発行事業者登録番号・振込先を記載） |
| 発注書（注文書＋注文請書） | `pdfkit`で項目リスト形式生成 | 顧客提供のExcelテンプレート（注文書・請書システムデモ用.xlsx）に準拠。工事名・工事内容・工事場所・工期・請負代金・支払条件の枠組み＋署名欄 |
| 請書PDF・請求書PDF（外部発行分） | ユーザーがアップロードした実PDFを保存 | `order_files`テーブルにbase64保存、モーダルでインラインプレビュー |

## 7. 運用・監視の現状と提案（未実装・検討中）

### エラー検知
- **現状**：APIエラーは`console.error`で吐き出すのみ→Vercelの関数ログに流れるが、**能動的な通知はなく、保持期間も短い**（見に行かないと気づけない）。
- **提案**：**Sentry**導入（バックエンド＋フロント）。エラー発生時のスタックトレース・頻度・影響ユーザーを可視化し、Slack等に通知可能。小規模利用なら無料枠（月5,000イベント）で足りる見込み。

### ログ保存
- **現状**：業務監査ログ（`audit_logs`）はDB永続化済みで問題なし。一方システムログ・実行ログはVercelの短期ログのみで**消えるリスク**がある。
- **提案**：Vercelの**Log Drain**機能で**Axiom**（またはBetter Stack）へ自動転送。Vercel公式連携で数クリック導入・無料枠あり。

### セキュリティ（検討事項）
- **IP制限**：Supabase Auth自体にはIP制限機能は無い（DBへの直接接続を制限する「Network Restrictions」は別機能でTeam/Enterprise向け）。本アプリで管理画面をIP制限したい場合は、**Vercel Firewall**（Pro以上）またはExpressミドルウェアでの自前実装が現実的。

## 8. 既知の制約・技術的注意点

| 項目 | 内容 |
|---|---|
| ファイルアップロード上限 | Vercelサーバーレス関数のリクエストボディ上限（約4.5MB）により、PDFは**3MBまで**に制限（base64化で約1.33倍に膨張するため）。恒久対応にはSupabase Storage等への直接アップロード方式への切替が必要。 |
| Tailwindビルド運用 | 新しいTailwindクラスを追加した際は`npm run build:css`の実行とコミットが必須（忘れるとスタイルが反映されない）。 |
| キャッシュバスティング | 共有JS（sidebar.js等）を更新した際は全HTMLの`?v=N`を一括で上げる必要がある（Safari等でキャッシュが残るため）。 |
| ブラウザ互換性 | `<input type="month">`や`<select>`のtext-align等、Safari特有の非対応挙動に複数回対応済み（date型への変更、text-align-lastの併用等）。 |
| macOS固有の開発環境問題 | プロジェクトディレクトリが`~/Desktop`配下にあるとmacOSのフォルダ保護により`npm`/`node`実行時に`EPERM uv_cwd`が発生するため、`~/construct-pro`（ホーム直下）に移設済み。 |
